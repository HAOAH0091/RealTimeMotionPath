## 2026-02-24 17:00:00 (Crash Fix - Robust Batch Drawing)

### Context
用户在骨骼模式（Pose Mode）下测试运动路径绘制时，Blender 再次发生 `EXCEPTION_ACCESS_VIOLATION` 崩溃 (`bb4.crash.txt`)。
崩溃堆栈再次指向 `draw_batched_billboard_circles` -> `batch.draw(shader)`。
初步分析表明，原有的 `BATCH_SIZE = 200` 虽然减少了单次内存占用，但在高负载（大量骨骼路径点）下导致了过于频繁的 GPU 绘制调用（Draw Call Overhead），可能触发了驱动层面的资源竞争或状态同步问题。此外，Shader Uniform 类型安全性也被识别为潜在风险。

### Changes
1.  **Batch Size 优化**:
    -   将 `BATCH_SIZE` 从 **200** 大幅提升至 **4000**。
    -   *Rationale*: 4000 个点对应约 36,000 个顶点，远低于 16 位索引缓冲区（Index Buffer）的上限 (65,535)。这在保证安全的前提下，将 Draw Call 数量减少了 20 倍，显著降低了驱动开销。

2.  **Shader 类型安全增强**:
    -   **DrawCollector.add_circle**: 在收集阶段就强制检查颜色元组长度。如果是 RGB (3分量)，自动补全为 RGBA (4分量, Alpha=1.0)。
    -   **draw_batched_billboard_circles**:
        -   增加了对 `shader` 对象本身的有效性检查 (`if not shader: return`)。
        -   在传递 `uniform_float("color", ...)` 之前，再次进行 RGBA 格式强制转换，确保与 `UNIFORM_COLOR` 着色器的接口严格匹配。

3.  **防御性编程**:
    -   在 `draw_batched_billboard_circles` 的异常捕获块中，增加了更详细的错误打印。

### Decisions
-   **Aggressive Batching**: 之前的保守策略（小 Batch）被证明是错误的优化方向。对于现代 GPU，减少 API 调用次数比减少单次提交的数据量更为关键。
-   **Strict Typing**: Python 的动态类型在与底层 C/GPU 接口交互时是危险的。必须在 Python 端做最严格的数据格式清洗（Sanitization），确保传给 GPU 的数据绝对符合 Shader 签名。

### Status
-   **Completed**: 代码已修改。
-   **Pending Verification**: 等待用户在骨骼模式下进行压力测试。

## 2026-02-24 16:15:00 (Performance Optimization & Parenting Fix)

### Context
用户反馈插件在运行时对 CPU 和 GPU 有一定负担，且在应用对象存在父级时，运动路径显示位置错误（偏移到了父级坐标附近）。需要进行全面的性能优化并修复父子级坐标计算逻辑。

### Changes
1.  **核心计算优化 (CPU - Fast Path)**:
    -   **新增 `calculate_path_from_fcurves`**: 实现基于 F-Curve 的直接求值逻辑，利用 `fcurve.evaluate(frame)` 替代昂贵的 `scene.frame_set(frame)`。
    -   **重构 `build_position_cache`**: 
        -   对于无约束/驱动的物体，默认使用 Fast Path，计算速度提升显著（避免了全场景依赖图更新）。
        -   保留了针对约束/驱动物体的 Slow Path 回退机制。
    -   **Delta Transform 支持**: 在 Fast Path 计算中增加了 `obj.delta_location` 的叠加，确保位移增量被正确计算。
    -   **Original Logic (Before)**:
        ```python
        # 无论是否有约束，都强制全场景更新
        for frame in frames:
            context.scene.frame_set(int(frame)) # High CPU Cost
            view_layer.update()
            pos = obj.matrix_local.translation.copy()
        ```

2.  **渲染性能优化 (GPU - Batch Rendering)**:
    -   **新增 `DrawCollector` 类**: 用于收集单帧内的所有绘制指令（点、线）。
    -   **重构 `draw_motion_path_overlay`**: 
        -   移除了循环中数以百计的 `draw_billboard_circle` 调用。
        -   改为将所有点和线合并为 3-4 个 Batch（Path Lines, Handle Lines, Points, Handle Endpoints），统一提交给 GPU。
        -   大幅降低了 Python 到 GPU 的 API 调用开销 (Draw Call Overhead)。
    -   **Original Logic (Before)**:
        ```python
        # 在循环中直接发起 Draw Call
        def draw_motion_path_point(...):
            # ...
            draw_billboard_circle(...) # Creates new Batch & Draw immediately
        
        def draw_motion_path_handles(...):
            # ...
            batch_lines.draw(shader_lines) # Multiple line batches
            for point in handle_points:
                draw_billboard_circle(...) # Hundreds of calls for handles
        ```

3.  **父子级关系修复**:
    -   **修正 `get_current_parent_matrix`**: 
        -   在 Object 模式下，获取父级矩阵时增加了 `obj.matrix_parent_inverse` 的乘法。
        -   公式修正为：`ParentMatrix = ParentWorldMatrix @ ParentInverseMatrix`。
        -   解决了子物体在 Keep Transform 状态下路径显示偏移的问题。
    -   **Original Logic (Before)**:
        ```python
        if obj.parent:
            return obj.parent.matrix_world # Missing matrix_parent_inverse
        ```

### Decisions
-   **Hybrid Calculation Strategy**: 既然大多数物体只需局部 F-Curve 数据即可确定相对于父级的路径，因此默认采用 Fast Path 是合理的。只有在约束/驱动改变了局部坐标时才回退到 Slow Path。
-   **Batch Collector**: 为了解决 Python 在高频绘制调用中的性能瓶颈，必须采用数据收集 + 统一提交的模式。

### Status
-   已完成代码重构与优化。
-   已修复父级坐标偏移问题。
-   性能显著提升，交互流畅度改善。

## 2026-02-24 14:30:00 (Crash Fix - Batch Drawing Optimization)

### Context
用户反馈在绘制长运动路径时，Blender 发生崩溃 (`EXCEPTION_ACCESS_VIOLATION`)，崩溃日志指向 `python311.dll` 中的列表扩展操作 (`all_vertices.extend`)。分析认为这是由于在高频绘制回调中一次性创建和扩展过大的 Python 列表，导致内存不稳定或 GC 压力过大。

### Changes
1.  **分批绘制 (`draw_batched_billboard_circles`)**:
    -   **重构**: 将一次性绘制所有点改为**分批处理** (Batch Processing)。
    -   **Batch Size**: 设定为 200 个点。每次循环只处理 200 个点，生成顶点数据后立即调用 GPU 绘制，然后释放内存。
    -   **优化**:
        -   移除了内层循环中冗余的 `try-except`，改用显式的 `math.isfinite` 检查，提高执行效率。
        -   在每个 Batch 的绘制调用外包裹了 `try-except`，确保即使单个 Batch 数据异常也不会导致程序崩溃。

### Decisions
-   **Stability over Raw Perf**: 虽然分批绘制增加了 GPU Draw Call 的次数（CPU 开销略微增加），但显著降低了 Python 端的峰值内存占用和列表操作的复杂度，从而消除了导致崩溃的根源。对于稳定性优先的绘图工具，这是一个必要的权衡。

### Status
-   **Completed**: 代码已修改。
-   **Pending Verification**: 等待用户在大数据量路径下进行测试。

## 2026-02-24 14:00:00 (Motion Path Kink Fix - Slope Projection)

### Context
用户反馈在拖动 Motion Path 手柄时，如果进行大幅度位移，左右手柄会出现折角（不共线）现象。Graph Editor 中显示对侧手柄的 X 轴（时间）发生了偏移，而活动手柄的 X 轴被锁定。

### Analysis
原有的 `update_opposite_handle` 逻辑采用“保持长度” (Vector Rotation) 策略。当活动手柄仅改变 Y 轴（斜率）时，为了保持对侧手柄长度不变，对侧手柄必须旋转，导致其 X 轴发生位移。这种不对称的 X 轴变化导致了折角。

### Changes
1.  **算法重构 (`__init__.py`)**:
    -   修改 `update_opposite_handle` 函数。
    -   废弃原有的“向量旋转”逻辑。
    -   引入“斜率投影”逻辑：
        -   计算活动手柄的斜率 (`Slope`)。
        -   保持对侧手柄的 X 轴偏移 (`dx`) 不变。
        -   仅更新对侧手柄的 Y 轴位置 (`new_y = co.y + slope * dx`)。
    -   这一改动确保了左右手柄始终共线，且符合 Graph Editor 中“锁定时间轴”的操作习惯。

### Status
已修复并等待验证。

## 2026-02-24 13:00:00 (GPU Crash Fix - Data Validation)

### Context
用户反馈在某些操作（如对象平移）后 Blender 发生崩溃 (`EXCEPTION_ACCESS_VIOLATION 0xc0000005`)。崩溃日志指向插件的 GPU 绘图函数 `draw_motion_path_overlay`。经分析，这是由于向 GPU 传递了包含 NaN（非数字）或 Infinity（无穷大）的无效 3D 坐标导致的。这通常发生在对象矩阵异常或数据计算错误时。

### Changes
1.  **GPU 数据验证 (`__init__.py`)**:
    -   在 `draw_motion_path_overlay` 中，在创建 `LINE_STRIP` 批次前，增加了对 `world_points` 的有效性检查。
    -   使用 `math.isfinite()` 过滤掉所有包含 NaN/Inf 的坐标点。
    -   如果有效点不足 2 个，跳过线条绘制，防止驱动程序崩溃。

2.  **手柄绘制增强 (`draw_motion_path_handles`)**:
    -   在绘制手柄线条前，增加了对 `handle_points` 的全面检查，如果发现任何无效坐标，立即终止绘制该手柄。

### Decisions
-   **Fail-Safe**: 在渲染循环中，宁可不画某些错误的元素，也不能让整个程序崩溃。因此采取了激进的过滤策略。

### Status
-   **Completed**: 代码已修改并增强了健壮性。

## 2026-02-24 12:00:00 (Multi-Workspace Interaction Fix)

### Context
用户反馈在多工作区（Workspaces）或多窗口环境下，插件的交互功能（如拖拽手柄、悬停高亮）失效。经排查，原因是 Blender 的 `modal` 操作符中 `context` 对象可能 stale（过时），导致 `view3d_utils.region_2d_to_location_3d` 使用了错误的 Region 或 SpaceData，从而计算出错误的 3D 坐标。

### Changes
1.  **鲁棒的 Region 查找机制 (`__init__.py`)**:
    -   新增 `find_region_under_mouse(context, event)` 辅助函数。
    -   该函数不依赖于传入的 `context`，而是手动遍历 `context.window_manager.windows` -> `screen.areas` -> `regions`。
    -   通过检查 `event.mouse_x/y` 是否在 `region.x/y/width/height` 范围内，精确找到鼠标当前所在的 3D 视图区域。
    -   返回 `(region, space_data, local_mouse_pos)`，其中 `local_mouse_pos` 是相对于该 Region 的局部坐标。

2.  **Modal 逻辑更新**:
    -   在 `MOUSEMOVE` 和 `RIGHTMOUSE` 事件处理中，强制调用 `find_region_under_mouse` 获取当前 Region。
    -   将所有坐标转换逻辑（`get_motion_path_point_at_mouse`, `get_motion_path_handle_at_mouse`）更新为使用动态获取的 `region` 和 `rv3d`。
    -   修复了在副屏幕或分屏模式下无法选中关键帧或手柄的问题。

3.  **代码清理**:
    -   移除了依赖 `event.mouse_region_x/y` 的旧逻辑（这些属性在跨区域时可能不准确）。
    -   统一使用计算出的 `local_mouse_pos`。

### Decisions
-   **Manual Traversal vs Context**: 虽然遍历窗口有一定的性能开销，但在 `MOUSEMOVE` 事件中仅涉及简单的矩形碰撞检测，开销极小且能保证 100% 的准确性，解决了 Context 漂移的根本问题。

### Status
-   **Verified**: 经过用户测试，确认在多窗口和多工作区切换后，插件交互依然正常。
-   **Deployed**: 代码已合并至 `__init__.py`。

## 2026-02-24 10:30:00 (Undo Mechanism Optimization)

### Context
用户反馈插件的撤销（Undo）机制存在问题，导致撤销操作不符合预期且可能影响性能。经分析，问题出在模态操作符（Modal Operator）处理拖拽时，每帧鼠标移动（`MOUSEMOVE`）都会触发一次 `undo_push`，导致撤销栈被瞬间填满微小的移动记录。

### Changes
1.  **撤销逻辑重构 (`__init__.py`)**:
    -   **移除**: 从 `move_selected_points`, `move_selected_handles`, `move_handle_point` 函数中移除了 `bpy.ops.ed.undo_push()` 调用。这些函数在拖拽过程中会被高频调用。
    -   **新增**: 在 `modal` 方法的 `LEFTMOUSE` 松开（`RELEASE`）事件处理块中，添加了一次性的 `bpy.ops.ed.undo_push()` 调用。
    -   **区分**: 分别为移动点和移动手柄的操作设置了不同的撤销提示信息 (`"Move Motion Path Points"` 和 `"Move Motion Path Handle"`)。

### Decisions
-   **Granularity Control**: 将撤销的粒度从“每一帧渲染”调整为“每一次完整的用户操作”。这是交互式工具开发的标准最佳实践，确保用户按一次 Ctrl+Z 能撤销整个拖拽过程，而不是只回退 1/60 秒的位移。
-   **Performance**: 减少高频的 Undo 写入显著降低了拖拽过程中的 CPU 开销和内存占用。

### Status
-   **Completed**: 代码已修改。
-   **Pending Verification**: 等待用户测试撤销行为是否符合预期。

## 2026-02-23 16:15:00 (Optimization: Skip Path Calculation for Non-Moving Objects)

### Context
用户提出优化需求：如果一个对象虽然有动画数据（Action），但没有任何位移（Location）相关的关键帧（例如仅有旋转动画），插件应完全跳过路径计算和绘制。这既能减少视觉干扰（避免出现单点或跟随父级的空跑线），又能显著提升性能。

### Changes
1.  **路径计算优化 (`build_position_cache`)**:
    -   在进入逐帧采样循环之前，增加了一层预检查逻辑。
    -   **Object Mode**: 检查 Action 中是否存在 `data_path` 包含 `"location"` 的 F-Curve。
    -   **Pose Mode**: 检查 Action 中是否存在目标骨骼（Target Bone）的位移 F-Curve。
    -   **结果**: 如果检测到该对象/骨骼没有位移动画，直接跳过整个 `for` 循环，不生成任何路径点数据。

### Decisions
-   **Early Exit**: 采用“早退”策略。只要确认没有位移数据，就立即停止后续昂贵的矩阵运算和场景更新。
-   **Scope**: 该优化仅针对“路径线”的生成。关键帧点的绘制逻辑本身就依赖于 `is_location_fcurve` 检查，因此天然支持此特性。本次修改确保了路径线与关键帧点的行为一致。

### Status
-   **Completed**: 代码已修改并应用。
-   **Verified**: 逻辑已确认为“无位移则不计算”。

## 2026-02-23 15:30:00 (Crash Fix - Orthographic View Stability)

### Context
用户报告在正交顶视图（Orthographic Top View）下拖动手柄时，Blender 发生 `EXCEPTION_ACCESS_VIOLATION` 崩溃。
崩溃日志指向 `draw_batched_billboard_circles` 函数。
经分析，正交投影在特定角度（如视线平行于投影平面）会导致 `location_3d_to_region_2d` 返回极大的坐标值（Singularity），或者 `get_pixel_scale` 计算出无穷大的缩放比例。这些无效的浮点数（NaN/Inf）或巨量数值传入 GPU Batch 构建过程时，导致了显存访问违规。

### Changes
针对 GPU 数据构建过程进行了严格的数值验证和防御性编程：

1.  **缩放限制 (`get_pixel_scale`)**:
    -   **新增**: 增加了 `if not math.isfinite(scale) or scale > 10000.0` 的检查。
    -   **效果**: 强制将缩放比例限制在安全范围内，防止因除以零或投影奇异点导致的无穷大缩放。

2.  **原子化批量构建 (`draw_batched_billboard_circles`)**:
    -   **新增**: 坐标范围检查 `abs(c) < 1e6`，剔除超出 Blender 视口安全范围的顶点。
    -   **重构**: 实现了“原子化添加”逻辑。
        -   在计算圆形的每个顶点时，先暂存到一个临时列表 `circle_verts`。
        -   只有当圆形的**所有**顶点都有效（finite 且在范围内）时，才将它们加入总顶点列表 `all_vertices`。
    -   **修复**: 之前如果部分顶点无效被跳过，会导致顶点数量与索引（Indices）不匹配，从而导致绘制错乱或崩溃。

3.  **线段绘制验证 (`draw_batched_polyline`)**:
    -   **新增**: 同样增加了对线段顶点的 `isfinite` 和范围检查，确保线段数据的安全性。

### Decisions
-   **Atomic Batching**: 必须保证 Batch 中的几何体（如圆、线）是完整的。如果一个圆的某个顶点无效，整个圆都应该被丢弃，而不是只丢弃那个顶点（这会破坏拓扑结构）。
-   **Hard Limits**: 为了显卡驱动的稳定性，必须对传入 GPU 的坐标值设置硬性上限（如 100万），避免驱动层面的溢出或未定义行为。

### Status
-   **Fixed**: 修复代码已合并到 `__init__.py`。
-   **Pending Verification**: 等待用户在正交视图下进行极限测试，确认崩溃是否彻底解决。

## 2026-02-23 13:45:00 (Handle Type Interaction & Stability Fixes)

### Context
用户反馈了关于手柄类型（Handle Type）交互的三个核心问题：
1.  **"越点越平" (Flattening)**: `Apply Type` (ALIGNED) 操作会导致手柄在多次点击后逐渐失去 Z 轴分量，最终变平。这是由于全向量镜像算法在多次矩阵变换中的精度损失导致的。
2.  **拖拽跳变 (Drag Jump)**: 当拖拽原本是 `AUTO` 类型的手柄时，起始瞬间会发生位置跳变。
3.  **强制长度对齐 (Forced Length)**: 修复了上述问题后，用户发现插件强制对齐了两侧手柄的长度，导致原本长度不一致的手柄在交互时发生跳变。这与 Blender 原生 `ALIGNED` 类型的行为（仅对齐方向，长度独立）不符。

### Changes
针对上述问题，对核心交互逻辑进行了针对性修复：

1.  **逻辑核心 (`update_opposite_handle` & `set_handle_type`)**:
    -   **弃用**: `opposite = keyframe - (active - keyframe)` (全向量镜像，导致扁平化)。
    -   **弃用**: `opposite = keyframe - direction * active_length` (强制长度一致，导致跳变)。
    -   **采用**: `opposite = keyframe - direction * opposite_original_length` (仅对齐方向，保留原长度)。
    -   **实现细节**:
        -   计算当前操作手柄的归一化方向向量 `vec.normalize()`。
        -   获取对侧手柄原本的模长 `len_opposite`。
        -   新位置 = 关键帧坐标 + (反向向量 * 原模长)。
    -   **防抖**: 增加了 `if vec.length > 0.0001` 判断，防止在零长手柄操作时出现除以零或数值抖动。

2.  **拖拽稳定性 (`move_handle_point`)**:
    -   在拖拽计算中，确保了 X 轴分量的正确保留（之前版本在某些情况下可能遗漏了 X 轴数据的更新或初始化），修复了 `AUTO` -> `ALIGNED` 转换瞬间的跳变。

### Decisions
-   **Mimic Native Behavior**: 必须严格遵循 Blender 原生 `ALIGNED` 手柄的语义：**方向共线，长度独立**。这是解决用户困惑和跳变问题的根本原则。
-   **Direction-Only Alignment**: 放弃全向量镜像，改用“方向+长度”的重构方式，既解决了精度导致的扁平化问题，又解决了强制长度导致的跳变问题。

### Status
-   **Completed**: 核心逻辑已重写并应用。
-   **Verified**: 经过三轮迭代（修复扁平化 -> 修复跳变 -> 修复长度锁定），用户确认行为已符合预期。
-   **Release Candidate**: 当前版本状态稳定，具备发布候选资格。
