# Smart Mode 对象锁定问题修复计划 (通俗版)

## 1. 深度分析与根本原因 (大白话解释)

### 现象描述
在 **Smart Mode**（智能模式）下，如果你的物体已经有了动画（比如做了位移），当你试图用鼠标 G 键移动它时，会发现它像被胶水粘住了一样，动弹不得，或者卡得要命。
但如果你把插件切换到 **Timer Mode**（定时器模式），哎？它又变正常了。

### 为什么会这样？（侦探故事版）

这里发生了一个 **"自己绊倒自己"** 的死循环事故。

**核心术语解释：**
*   **Depsgraph (依赖图)**: 相当于 Blender 的"情报中心"。一旦场景里有东西变了（比如你移动了方块），它就会广播："注意啦！数据变了！"
*   **Handler (处理程序)**: 我们的插件派出的"监听员" (`on_depsgraph_update`)。它一听到广播，就立刻去干活（重新计算路径）。
*   **frame_set (切换帧)**: 相当于在时间轴上跳跃。比如从第1帧跳到第10帧。**重点是：这个动作本身也会触发"情报中心"的广播！**
*   **Recursion (递归/死循环)**: A 叫 B 干活，B 为了干活又叫 A，A 又叫 B……无穷无尽。

**事故现场还原：**

1.  **你动了一下手**: 你用 G 键移动了物体。
2.  **情报中心广播**: "物体动了！"
3.  **监听员出动**: 我们的插件听到了，心想："好，我得重新画路径线。"
4.  **开始干活**: 插件开始计算。为了画出准确的线，它需要知道物体在每一帧的位置。
5.  **危险动作**: 插件说："我得去第10帧看看。" 于是它执行了 `frame_set(10)`。
6.  **再次广播**: 就在它切到第10帧的一瞬间，情报中心又喊："注意！时间变了！数据变了！"
7.  **监听员懵了**: 我们的监听员（还是同一个）听到广播，条件反射地想："又有变化？那我得重新计算！"
8.  **死循环**:
    *   第一次计算还没结束呢，第二次计算又开始了。
    *   第二次计算里，它又要切帧，于是又触发第三次……
    *   电脑瞬间被成千上万次的计算请求塞满了，CPU 冒烟，你的鼠标操作根本挤不进去。
    *   这就导致了你感觉到的 **"锁死"**。

### 为什么 Timer Mode 没问题？
Timer Mode 就像个 **"定闹钟的懒人"**。它不管情报中心怎么喊，它只看表：
"现在是 10:00:00，我算一次。"
"现在是 10:00:01，我再算一次。"
就算它计算过程中切了帧，引发了广播，它也不理会（因为它只听闹钟的）。等它算完了，休息够了（0.1秒后），才进行下一次。所以它不会死循环。

---

## 2. 修复方案：给插件装个"忙碌牌" (Atomic Lock)

我们要彻底解决这个问题，就要让插件变得更聪明，学会**"一次只做一件事"**。

### 核心策略：原子锁 (Atomic Lock)

我们要在计算函数 (`build_position_cache`) 的大门口挂一个牌子：**"计算中，请勿打扰"**。

**修复后的流程：**

1.  **你动了一下手** -> 情报中心广播。
2.  **监听员出动**，试图调用计算函数。
3.  **看门大爷 (锁检查)**:
    *   计算函数先看一眼门口的牌子。
    *   牌子是空的？好，挂上 **"计算中"**，进去干活。
4.  **危险动作**: 里面开始切帧 `frame_set(10)`。
5.  **再次广播**: 情报中心又喊了。
6.  **监听员又来了**: 试图再次调用计算函数。
7.  **看门大爷拦截**:
    *   一看门口挂着 **"计算中"**。
    *   大爷直接说："里面正忙着呢，你这个请求无效，回去吧！"
    *   **死循环被打断了！**
8.  **任务完成**: 第一次计算结束，摘下牌子。一切恢复平静。

### 实施步骤
我们将修改代码，把这个"挂牌子"的动作，从原来的"监听员"那里，挪到"计算函数"的最里面。这样无论谁来调用，只要里面在忙，都能被挡回去。

## 3. 验证方法
修复后，你在 Smart Mode 下移动物体，应该会像 Timer Mode 一样流畅，不再卡顿。
