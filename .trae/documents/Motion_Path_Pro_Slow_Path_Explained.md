# Slow Path 为什么不能简单地"不动"？(大白话解释)

您问得很好：**既然 Fast Path 可以靠"只读旧数据"来避开交互冲突，为什么 Slow Path 不行？**

这就涉及到了它们俩**数据来源**的根本区别。

### 1. Fast Path (普通对象)
*   **数据来源**: **剧本 (F-Curves)**。
*   **比喻**: 就像一个演员（物体），它的所有走位都写死在剧本（曲线）里了。
*   **交互时**: 哪怕导演（您）现在正拽着演员满场跑（G键移动），只要没喊"Cut"（松开鼠标确认），剧本本身是没变的。
*   **插件做法**: 我不管你在场上怎么折腾，我只看手里的剧本。所以我能画出原来的路线，而且因为我不上场干涉，你也感觉不到我。

### 2. Slow Path (带约束/驱动的对象)
*   **数据来源**: **现场彩排 (Scene Evaluation)**。
*   **比喻**: 这个演员（物体）是"身不由己"的。比如它被绳子拴在另一个演员身上（父子约束），或者它的位置是由复杂的机关控制的（驱动器）。它的位置没有写在剧本里，而是**算出来的**。
*   **交互时**: 如果你想知道它在第10秒在哪，你不能看剧本（因为剧本没写），你必须让所有演员真的走到第10秒的位置，摆好姿势，量一下绳子长度，才能算出它在哪。
*   **冲突点**:
    *   **您的操作**: 您正在第1秒拽着它。
    *   **插件需求**: 插件为了画线，必须命令场景："所有人！立刻跳到第2秒！第3秒！第4秒！...摆好姿势让我拍照！"
    *   **打架了**:
        *   插件大喊："切到第2秒！" -> 场景变了。
        *   您的 G 键操作："哎？我刚才拽到一半的手感怎么没了？" -> 因为场景被插件强行重置了。
        *   如果不切回去（`frame_set`），场景就停在第100帧了，您的操作就更乱了。
        *   如果切回去（`frame_current`），那就是一次完整的"破坏现场再恢复现场"，您的交互状态（G键的临时位移）在这个过程中就被**冲刷**掉了。

### 总结
*   **Fast Path**: 可以"纸上谈兵"（只读数据），所以不干扰现场。
*   **Slow Path**: 必须"实地演习"（切帧计算），这必然会打断您当前的"现场操作"。

### 那么 Slow Path 该怎么办？
既然无法共存，最好的策略就是 **"避让"**。
如果检测到您正在交互（比如 G 键移动中），Slow Path 应该**完全停止计算**，连"旧数据"都不算（因为没法算，除非有缓存）。等您松手了，它再重新跑一遍"实地演习"。
目前的代码虽然没有显式的"避让"逻辑，但加上了"原子锁"后，至少能保证它不会无限死循环，只是可能会感觉卡顿（因为通过切帧来计算本身就很重）。

**最简单的优化**:
如果不希望 Slow Path 干扰交互，我们可以加一个简单的判断：
**"如果我在 Slow Path 模式下，且检测到当前正在播放动画或频繁更新，我就直接罢工不干了，直到世界安静下来。"**
但鉴于您目前的问题主要在 Fast Path 上已经解决，Slow Path 的优化可以作为未来的锦上添花。
