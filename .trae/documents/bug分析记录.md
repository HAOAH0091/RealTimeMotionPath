# Motion Path Pro 手柄Bug分析记录

## 📋 问题概述
### 问题 1（已部分修复）：父级变换后手柄偏移
当父级物体（或骨骼）进行旋转、缩放、移动等变换时，运动路径上的手柄方向会发生偏移，不再与运动路径相切。

### 问题 2（新发现）：对象自身旋转后手柄也旋转
当对象（或骨骼）自身进行旋转、缩放时，运动路径上的手柄也会跟着旋转，这是不对的！
- **预期行为**：手柄应该反映F-Curve的方向，不受对象自身变换影响
- **实际行为**：对象自身旋转后，手柄也跟着旋转了

## 🎯 测试场景
```
空物体（父级）→ 旋转 -90° 绕Z轴
    └── 锥体（子级）→ 有位置动画
```

### 现象
1. ✅ 默认值下（无父级变换）：手柄正常，与运动路径相切
2. ❌ 父级旋转 -90° 后：手柄方向明显偏移
3. ❌ 骨骼模式下也存在同样问题

## 🔬 代码分析

### 问题 0：新发现的核心问题 - 对象自身变换也影响了手柄

#### 根本原因
`obj.matrix_world` 包含了**两部分**变换：
1. ✅ 父级的变换（需要保留）
2. ❌ 对象自身的变换（需要排除）

#### 数学原理
我们需要的是**只包含父级变换**的矩阵，方法：
```python
父级变换矩阵 = obj.matrix_world @ obj.matrix_local.inverted()
```

### 问题 1：对象模式下的致命错误

**文件位置：** `__init__.py`，函数 `draw_motion_path_handles()` 行 358-359

**当前代码：**
```python
else:
    rotation_matrix = mathutils.Matrix.Identity(3)  # ❌ 错误！
```

**问题说明：**
- `point_3d` 使用了 `obj.matrix_world.translation`，正确包含了父级变换
- 但手柄向量的转换矩阵直接用了单位矩阵，完全没有考虑 `obj.matrix_world`
- 结果：关键帧位置对了，但手柄向量没旋转

**应该改为：**
```python
else:
    rotation_matrix = obj.matrix_world.to_3x3()  # ✅ 正确！
```

### 问题 2：骨骼模式下的潜在不一致

**对比两个函数：**

| 函数 | 位置 | 使用的矩阵 |
|------|------|-----------|
| `draw_motion_path_handles` | 行 354 | `obj.matrix_world @ bone.matrix` |
| `get_motion_path_handle_at_mouse` | 行 1142 | `obj.matrix_world @ parent.matrix` |

**潜在问题：**

- `draw` 用的是 `bone.matrix`（当前骨骼的完整矩阵）
- `get_motion_path_handle_at_mouse` 用的是 `parent.matrix`（父骨骼的矩阵）

## 📊 核心概念澄清

### 1. F-Curve 存储的是什么？
✅ F-Curve 存储的是**局部坐标**（相对于父级的坐标）

### 2. 向量 vs 点的区别

| 概念 | 是什么 | 例子 | 受位移影响吗？ |
|------|--------|------|---------------|
| **点（Point）** | 一个位置 | 关键帧在哪里 | ✅ 受影响 |
| **向量（Vector）** | 方向和长度 | 手柄往哪个方向伸 | ❌ 不受影响 |

### 3. `to_3x3()` 的作用
`obj.matrix_world` 是 4x4 矩阵：
```
[ R R R T ]
[ R R R T ]
[ R R R T ]
[ 0 0 0 1 ]
```
- **R** = 旋转缩放部分（左上角 3x3）
- **T** = 位移部分（最右边一列）

`.to_3x3()` 只提取 **R** 部分，丢弃 **T** 部分——这正是我们想要的，因为向量不需要位移！
