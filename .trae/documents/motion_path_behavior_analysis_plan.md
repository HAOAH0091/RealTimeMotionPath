# 功能逻辑确认与性能分析报告

## 1. 现状确认

### 问题 1：如果一个对象没有位移动画，还会为它绘制吗？
这取决于对象是否有“动画数据块 (Action)”：
*   **情况 A：对象完全没有动画数据**（即没有 Action）：
    *   **不会绘制**。插件会直接跳过，不进行任何计算。
*   **情况 B：对象有动画数据，但没有“位移 (Location)”关键帧**（例如只有旋转动画）：
    *   **关键帧点**：不会绘制（插件会过滤掉非 Location 的曲线）。
    *   **运动路径线**：**会绘制**。插件依然会逐帧扫描该对象的位置。
        *   如果对象相对于世界坐标静止（只是原地旋转），你会看到一个点。
        *   如果对象跟随父级移动，你会看到它随父级运动的轨迹线。

### 问题 2：切换选中对象后，数据是否保留？
*   **不保留，会重新计算**。
*   **机制**：插件的设计是针对“当前激活对象 (Active Object)”进行服务的。
*   **流程**：当你选中对象 A -> 插件计算 A 的路径 -> 选中对象 B -> 插件清空 A 的数据，计算 B 的路径 -> 再次选中 A -> 插件**重新**逐帧计算 A 的路径。
*   **原因**：目前的缓存机制 (`_state.position_cache`) 是临时的，且只针对当前对象。

## 2. 性能负担分析

**结论：是的，会有显著的性能负担。**

*   **计算成本高**：
    *   为了绘制平滑的连续路径线，插件目前采用的是**逐帧采样 (Frame-by-Frame Sampling)**。
    *   代码逻辑：`for f in range(start, end): context.scene.frame_set(f)`。
*   **场景越复杂，卡顿越明显**：
    *   `frame_set(f)` 会强制 Blender 更新整个场景的依赖图（包括骨骼解算、修改器、物理等）。
    *   如果您的动画有 100 帧，每次您重新选中这个对象，Blender 就必须在后台瞬间运算 100 次场景状态。
    *   在简单场景中可能只是“微卡”，但在复杂角色或大场景中，这会导致明显的界面冻结（Lag）。

## 3. 优化建议（可选）

如果您觉得当前的性能影响了工作流，我们可以考虑以下优化方案：

*   **方案 A：增加对象级缓存（推荐）**
    *   **原理**：将计算好的路径数据保存在一个全局字典中，以“对象名”为索引。
    *   **效果**：切换回已计算过的对象时，直接读取缓存，**零延迟**显示。只有在检测到该对象的动画被修改时才重算。
    
*   **方案 B：限制采样密度**
    *   **原理**：不每帧都算，改为每隔 N 帧算一次（线性插值），或者只在关键帧附近密集采样。
    *   **效果**：降低计算量，但路径可能会变“直”，不够圆滑。

目前我们将维持现状，除非您希望执行优化。
