# 终极简化方案：检测并跳过交互

您提出了一个非常好的简化思路：**既然我们无法在交互期间完美更新路径，那就不要更新！**

这比试图在每一帧都精准更新要稳健得多。
只要我们能检测到用户正在移动对象（交互中），我们就不做任何计算。等用户松手了，我们再计算一次。

### 核心问题：如何检测交互？

Blender 的 Python API 有一个缺陷：它很难直接知道"当前是否有一个 G/R/S 操作符正在运行"。
但是，我们有几个间接的办法：

1.  **检查 `depsgraph` 的更新标志**
    *   当用户按 G 键移动时，`depsgraph.id_type_updated('OBJECT')` 会持续为 True。
    *   但这无法区分是脚本改的还是鼠标改的。

2.  **检查 `context.mode`**
    *   `G` 操作符也是在 `OBJECT` 模式下运行的，所以无法区分。

3.  **检查 `bpy.context.workspace.status_text`** (不推荐，不稳定)

4.  **利用 `is_evaluated` 属性**
    *   当我们在 `depsgraph_update_post` 回调中时，场景已经被评估过了。
    *   如果是 G 操作触发的，此时对象的位置是临时的。

### 换个思路：我们真的需要检测吗？

其实，如果我们在 `build_position_cache` 中执行的是**纯读取操作**（Fast Path），它本身是安全的，**只要不碰 `frame_set` 和 `view_layer.update`**。
您提到的"关键帧之间移动，位置是临时的"，这正好！
如果我们只是读取曲线数据（F-Curves），那么无论对象现在被移到哪里，**曲线数据还是旧的**（除非开启了自动插帧且当前就在关键帧上）。
所以，如果我们只读曲线，算出来的路径就是**原来的路径**。这正是您想要的！

**结论**：
只要我们严格遵守 **"只读不写"** 原则（Fast Path），我们甚至不需要检测交互。
1.  用户按 G 移动 -> 对象位置变了（临时）。
2.  触发更新 -> 插件读取 F-Curves（旧数据）。
3.  插件画出原来的路径。
4.  用户确认移动（点击鼠标） -> 写入关键帧（或自动插帧）。
5.  触发更新 -> 插件读取 F-Curves（新数据）。
6.  插件画出新路径。

这完全符合逻辑！

### 为什么之前会"锁死"？
就是因为我之前的代码在函数末尾手欠加了一句 `view_layer.update()`。
这句代码的意思是："Blender，请忽略所有临时状态，强制刷新！"
于是 G 操作的临时状态被强制刷新没了。

### 修复方案 (Simple & Clean)

我们不需要复杂的交互检测逻辑。我们只需要：
**删除 `build_position_cache` 末尾的 `view_layer.update()` 和 `scene.frame_current` 恢复逻辑（针对 Fast Path）。**

这就够了。
1.  **Fast Path**: 纯数学计算，不碰场景，不碰帧。G 操作随意进行，路径保持不变（直到确认）。
2.  **Slow Path**: 必须切帧。这在交互时确实会冲突。对于这种情况，我们可以加一个简单的锁：如果检测到 `depsgraph` 是由 TRANSFORM 引起的（虽然很难检测），或者干脆接受 Slow Path 对象在交互时会重置（Blender 限制）。但对于您的 Cone（Fast Path），问题将完美解决。

### 关于"自动插帧"
如果用户开启了自动插帧，移动过程中会实时修改 F-Curve 吗？
通常是在确认（Release）的那一瞬间修改。
所以在移动过程中，F-Curve 保持不变，路径也保持不变。这正是您期望的"先不动，等完了再动"。

**行动计划**：
只做减法。移除那些破坏性的状态重置代码。
