# 手柄与路径不相切问题深度分析

## 1. 当前现象
用户反馈在**父级存在旋转动画**的情况下，子物体（圆锥）的运动路径手柄与实际的运动路径（黑色虚线）**不相切**（方向不一致）。

![问题示意](image_placeholder)
*(根据用户描述：父级旋转动画导致手柄方向与路径分离)*

## 2. 当前代码逻辑分析

### 2.1 绘制流程 (`draw_motion_path_handles`)
目前插件绘制手柄的逻辑如下：
1.  **获取数据**：从 F-Curve 中读取关键帧的**局部**手柄坐标（`handle_left`, `handle_right`）。
2.  **计算局部向量**：计算局部空间下的手柄方向向量 `Vector_Local`。
3.  **获取父级变换**：获取当前帧（或关键帧时刻）父级到世界的变换矩阵 `Parent_Matrix`。
4.  **空间转换**：
    ```python
    Rotation_Matrix = Parent_Matrix.to_3x3()  # 提取旋转和缩放
    Vector_World = Rotation_Matrix @ Vector_Local
    ```
5.  **绘制**：在世界坐标中绘制这个 `Vector_World`。

### 2.2 交互流程 (`move_handle_point`)
当用户拖动手柄时：
1.  **获取世界向量**：计算用户鼠标拖动形成的新世界向量 `New_Vector_World`。
2.  **逆向转换**：
    ```python
    Rotation_Matrix = Parent_Matrix.to_3x3()
    New_Vector_Local = Rotation_Matrix.inverted() @ New_Vector_World
    ```
3.  **更新数据**：将 `New_Vector_Local` 写入 F-Curve。

## 3. 问题根源：数学原理

这个问题的根本原因在于：**"变换后的局部速度" 不等于 "世界空间的总速度"**。

### 3.1 速度合成公式
物体在世界空间的位置 $P_{world}(t)$ 由父级变换 $M_{parent}(t)$ 和局部位置 $P_{local}(t)$ 决定：
$$ P_{world}(t) = M_{parent}(t) \times P_{local}(t) $$

运动路径的切线方向（即世界速度 $V_{world}$）是对时间 $t$ 求导：
$$ V_{world} = \frac{d(P_{world})}{dt} = \underbrace{M_{parent}(t) \times \frac{d(P_{local})}{dt}}_{\text{项 A: 局部运动贡献}} + \underbrace{\frac{d(M_{parent})}{dt} \times P_{local}(t)}_{\text{项 B: 父级运动贡献}} $$

### 3.2 插件当前表现
*   **手柄显示的是**：**项 A**（$M_{parent}(t) \times V_{local}$）。
    *   插件将 F-Curve 中的局部切线仅仅做了旋转变换。
    *   这代表了"如果父级瞬间静止，子物体的运动方向"。
*   **路径显示的是**：**$P_{world}(t)$ 的轨迹**。
    *   路径的切线是 **$V_{world}$（项 A + 项 B）**。
*   **差异来源**：**项 B**（父级运动带来的牵连速度）。
    *   当父级有**旋转动画**或**位移动画**时，$\frac{d(M_{parent})}{dt}$ 不为零。
    *   父级的旋转会给子物体施加一个切向速度（类似于旋转木马上的离心/切向速度）。
    *   **这就是为什么手柄（项A）不指向路径切线（项A+项B）的原因。**

### 3.3 举例说明
想象你在一个旋转的圆盘（父级）上向外走直线（子级局部运动）：
*   **你的局部感觉**：我在走直线（手柄指向直线方向）。
*   **外部观察者（世界视角）**：你的轨迹是一条螺旋线（路径是弯曲的）。
*   **当前的插件显示**：手柄指向你走直线的方向，但路径是螺旋的。所以手柄看起来和路径"分叉"了。

## 4. 结论与权衡

这**不是代码写错了**，而是**可视化目标的选择**问题。

### 方案 A：保持现状（显示局部控制方向）
*   **含义**：手柄代表 F-Curve 中存储的真实数据方向。
*   **优点**：**所见即所得（数据层面）**。手柄方向直观对应 Graph Editor 中的曲线斜率。拖动手柄直接改变局部运动。
*   **缺点**：**视觉不直观**。在父级运动剧烈时，手柄不与路径相切，用户会觉得"坏了"。

### 方案 B：显示世界切线（显示最终运动方向）
*   **含义**：计算并显示 $V_{world}$。
*   **优点**：**视觉完美**。手柄永远与路径相切。
*   **缺点**：**交互复杂且容易产生误导**。
    *   如果用户把手柄拽到指向 X 轴，系统必须计算 $V_{local} = V_{world} - V_{parent}$。
    *   在某些情况下（例如父级速度极快），用户可能无法通过调整局部速度来达到想要的世界方向（数学上的限制）。
    *   F-Curve 的形状可能与视口中看到的手柄完全不同。

## 5. 建议
鉴于这是一个用来编辑 F-Curve 的工具，**方案 A（保持现状）**通常是更严谨的做法，因为我们编辑的是"局部动画"。

但是，为了减少用户的困惑，我们可以考虑：
1.  **确认父级矩阵获取的准确性**：确保我们使用的是当前子级关键帧时刻的**精确**父级矩阵（目前已经是这样）。
2.  **UI/UX 提示**：或许可以在文档中说明，当父级有动画时，手柄显示的是"相对运动方向"，而非"绝对轨迹切线"。

如果用户强烈要求手柄必须切于路径，我们需要引入复杂的物理速度合成计算，但这可能会让 F-Curve 的编辑变得不可预测。
