# 🐛 Bug分析：末尾关键帧手柄消失及无法拖动问题

## 1. 问题现象
用户反馈在操作运动路径时，**末尾关键帧**出现以下异常：
1.  **手柄消失**：看不见左右手柄的控制点和连线。
2.  **无法拖动位置**：尝试拖动该关键帧时，关键帧位置（圆点）不移动，但运动路径的形状会发生变化（说明产生了某种交互）。
3.  **手柄被"锁死"在关键帧上**：即使用户尝试盲操作拖动手柄，虽然路径形状改变了（说明F-Curve数据变了），但视觉上的手柄依然死死地钉在关键帧点上，无法被拉开。

## 2. 深度原因分析

经过对 `__init__.py` 代码的详细审查，发现这是一个**视觉逻辑与数据脱钩**导致的复合问题。

### 2.1 为什么手柄消失了？（且被锁定位置）
问题出在 `draw_motion_path_handles` 函数中的**视觉矫正逻辑**（Visual Correction）。

-   **逻辑回顾**：为了让手柄与运动路径视觉上相切，我们引入了 `world_velocity`（世界速度）来计算手柄的世界坐标：
    ```python
    # 伪代码
    dt = handle_frame - keyframe_frame  # 时间差（通常是 -1 或 +1）
    handle_pos = point_3d + (world_velocity * dt) * global_scale
    ```
-   **计算隐患**：`world_velocity` 是通过有限差分法计算的（`pos_next - pos_prev`）。
-   **末尾帧的特殊性**：在场景或动作的**最后一帧**（例如 frame 120），`context.scene.frame_set(frame + 0.01)` 可能会因为**帧范围限制**（Clamping）而被钳制在 120 帧。
    -   导致 `pos_next` (120.01) 与 `pos_current` (120) 位置完全相同。
    -   如果 `pos_prev` 也因某种原因（如曲线平缓或插值方式）接近当前位置，或者计算出的 `world_velocity` 极其微小甚至为零。
-   **致命后果**：当 `world_velocity` 为 `(0, 0, 0)` 时：
    -   **公式变成了**：`handle_pos = point_3d + (0 * dt) * global_scale`
    -   **结果恒等于**：`handle_pos = point_3d`
    -   **这意味着**：无论 F-Curve 中的手柄值（Value）如何变化，只要 `dt`（时间差）不变，**手柄的绘制位置永远等于关键帧的位置**。
    -   所以用户看到的不是"手柄消失了"，而是手柄**长度被强制归零**并重合在关键帧上。

### 2.2 为什么关键帧位置无法拖动？
这是一个由上述"视觉锁定"导致的**交互死循环**。

-   **交互优先级**：在 `MOTIONPATH_DirectManipulation.modal` 中，交互检测的顺序是：
    1.  **检测手柄** (`get_handle_point_at_mouse`) —— **优先级高**
    2.  **检测路径点** (`get_motion_path_point_at_mouse`) —— **优先级低**

-   **死循环过程**：
    1.  **视觉重合**：由于速度为0，手柄绘制位置被迫与关键帧点重合。
    2.  **误判操作**：用户点击关键帧点，程序优先检测到手柄，进入"手柄拖动模式"。
    3.  **数据更新**：用户拖动鼠标 -> F-Curve 手柄数据更新 -> 路径形状改变（符合预期）。
    4.  **视觉未更新**：虽然 F-Curve 变了，但因为**速度依然计算为0**，下一帧绘制时手柄**依然被强制画在关键帧点上**。
    5.  **结果**：用户永远无法把手柄"拉开"以露出下面的关键帧点，导致永远无法选中并拖动关键帧本身。

## 3. 结论
这个问题的根源在于**末尾帧的速度计算失效导致手柄长度为零**，进而导致**手柄与点重合**，最终引发了**交互目标的误判**。

## 4. 修复方案

### 核心修复：完善手柄绘制逻辑
在 `draw_motion_path_handles` 中增加**回退机制**（Fallback Mechanism）。

**修改前**：
只要 `world_velocity` 存在（非 None），就强制使用速度法计算手柄位置。

**修改后**：
检查 `world_velocity` 的长度。如果速度过小（接近零），说明速度法失效（可能是静止物体或末尾帧钳制），此时应**回退到原始逻辑**——即直接使用 F-Curve 的手柄数据并乘以父级旋转矩阵。

这将确保：
1.  即使在末尾帧，手柄也能根据 F-Curve 数据正确显示出来（不再长度为 0）。
2.  手柄不再与关键帧重合，交互检测将恢复正常。
3.  用户既可以看到手柄，也能正常拖动关键帧。
