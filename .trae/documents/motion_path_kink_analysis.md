# 问题分析：手柄折角原因与解决方案

## 1. 现象确认
您在 3D 视图中观察到，当大幅度调整手柄时，左右手柄出现了明显的**折角（Kink）**，不再保持一条直线。同时 Graph Editor 中的曲线也显示手柄不再对齐。

## 2. 原因分析
经过代码分析，这个现象由以下两个原因共同导致：

### 核心原因：手柄类型转换 (Vector -> Free)
1.  **初始状态**：您的关键帧手柄类型最初可能是 **Vector (矢量)** 或 **Free (自由)**。
2.  **代码逻辑**：在 `move_selected_handles` 函数中，为了允许您自由拖动手柄，代码会自动检测：如果是 `VECTOR` 类型，会强制转换为 `FREE` 类型。
    ```python
    # 现有代码逻辑
    self.convert_vector_handles_to_free(keyframes_for_location) # 强制转为 Free
    ```
3.  **对齐失效**：一旦变为 `FREE` 类型，代码中的 `update_opposite_handle` 函数就会**停止工作**（因为它只对 `ALIGNED` 类型生效）。
    *   结果：您拖动的一侧手柄移动了，但另一侧手柄保持不动。
    *   **结论**：这就是产生折角的直接原因。

### 次要原因：视觉误差 (Parent Velocity)
*   **现象**：即使没有折角，手柄有时看起来也没有指向运动路径（白线）的方向。
*   **原因**：目前的绘制代码只考虑了“局部手柄旋转到世界空间”，而**没有叠加父级物体的运动速度**。当父级物体移动时，真实的运动路径是“父级运动 + 子级运动”的合成，而手柄只显示了“子级运动”。这会导致手柄在视觉上与路径“脱节”。

## 3. 解决方案建议

为了解决这个问题，我建议进行以下优化：

### 方案 A：优化交互逻辑（推荐）
修改 `convert_vector_handles_to_free` 的行为，或者在拖动时提供选项：
*   **改动**：当拖动 `VECTOR` 手柄时，将其转换为 **`ALIGNED` (对齐)** 而不是 `FREE`。
*   **效果**：这样在拖动一侧时，另一侧会自动反向旋转，始终保持平滑，不会出现折角。

### 方案 B：引入父级速度修正 (Visual Correction)
*   **改动**：在绘制手柄时，计算父级在当前帧的速度，并叠加到手柄显示上。
*   **效果**：手柄将完美贴合白色的运动路径曲线，实现“所见即所得”。
