# Motion Path Pro 插件技术解析报告

## 1. 概述
本插件旨在弥补 Blender 原生运动路径功能的不足，提供**实时的**、**可交互的**、**可视化的**运动路径编辑功能。它允许用户在 3D 视图中直接看到关键帧的控制手柄（Handle），并对其进行拖拽编辑，从而直观地调整动画曲线，而无需频繁切换到图形编辑器（Graph Editor）。

## 2. 核心原理：数据生成与缓存 (The Brain)

插件并不会重绘整条运动轨迹线（这部分通常依赖 Blender 原生功能），它的核心在于**生成和绘制关键帧处的控制手柄**。

### 2.1 为什么需要自定义计算？
Blender 的 F-Curve 存储的是**局部空间 (Local Space)** 的数据。当物体有父级（Parent）或者约束（Constraint）时，物体在**世界空间 (World Space)** 的实际运动轨迹会受到父级运动的影响。
单纯读取 F-Curve 的手柄数据并不能直接得到世界空间中的正确切线方向。因此，插件需要计算物体在世界空间中的**真实速度向量**。

### 2.2 核心算法：有限差分法 (Finite Difference)
为了获取物体在某关键帧处的“真实世界速度”，插件使用了**有限差分法**。

*   **代码位置**: `build_position_cache` 函数
*   **逻辑**:
    1.  **采样前一刻**: 将时间设为 `当前帧 - 0.01`，获取物体世界坐标 `P_prev`。
    2.  **采样当前刻**: 将时间设为 `当前帧`，获取物体世界坐标 `P_curr`。
    3.  **采样后一刻**: 将时间设为 `当前帧 + 0.01`，获取物体世界坐标 `P_next`。
    4.  **计算速度**:
        *   入速度 (In-Velocity) ≈ `(P_curr - P_prev) / 0.01`
        *   出速度 (Out-Velocity) ≈ `(P_next - P_curr) / 0.01`

这种方法能够完美捕捉到父级旋转、缩放以及各种约束对物体运动方向的影响，确保绘制出的手柄永远与实际运动轨迹**相切**。

### 2.3 数据缓存结构
所有计算结果被存储在全局变量 `_state.position_cache` 中，结构如下：
```python
cache = {
    "Bone_Name_or_None": {
        frame_number: {
            'position': Vector(...),       # 关键帧世界坐标
            'parent_matrix': Matrix(...),  # 父级变换矩阵（用于坐标转换）
            'velocity_prev': Vector(...),  # 入射速度（用于左手柄方向）
            'velocity_next': Vector(...)   # 出射速度（用于右手柄方向）
        }
    }
}
```

## 3. 核心原理：绘制实现 (The Eyes)

插件使用 Blender 的 `gpu` 模块进行高性能绘制。绘制是在 3D 视图的覆盖层（Overlay）中进行的。

### 3.1 绘图入口
通过 `bpy.types.SpaceView3D.draw_handler_add` 注册回调函数 `draw_motion_path_overlay`。这意味着每当 3D 视图刷新时，插件的代码就会运行一次以绘制图形。

### 3.2 关键技术：广告牌技术 (Billboarding)
为了让关键帧点和手柄控制点在 3D 空间中始终面向摄像机（看起来像 2D UI 元素），插件使用了广告牌技术。

*   **代码位置**: `draw_billboard_circle` 和 `draw_billboard_square`
*   **原理**:
    1.  获取摄像机的视图旋转矩阵 `rv3d.view_rotation`。
    2.  计算摄像机的 `Right`（右）和 `Up`（上）向量。
    3.  以 3D 点为中心，利用 `Right` 和 `Up` 向量偏移顶点，构建出一个始终垂直于视线的平面（圆形或方形）。
    4.  **动态缩放**: 利用 `view3d_utils.location_3d_to_region_2d` 计算透视缩放比例，确保无论摄像机远近，屏幕上的点大小（像素）保持一致。

### 3.3 手柄绘制逻辑
手柄的绘制结合了**世界空间方向**和**局部空间长度**。

*   **方向 (Direction)**: 由 2.2 节计算出的 `velocity` 决定。这保证了手柄视觉上与路径相切。
*   **长度 (Length)**: 读取 F-Curve 中的 `handle_left` / `handle_right` 与 `co`（关键帧坐标）的差值，经过父级矩阵变换到世界空间，计算出长度。
*   **绘制**: 使用 `gpu.shader.from_builtin('SMOOTH_COLOR')` 绘制线段，连接关键帧点和手柄末端。

## 4. 核心原理：坐标变换 (The Bridge)

这是插件最复杂的部分，负责处理“看”与“改”之间的矛盾。

*   **看 (Draw)**: `F-Curve (Local)` -> `Parent Matrix` -> `World Handle`
*   **改 (Edit)**: `World Mouse Drag` -> `Inverse Parent Matrix` -> `F-Curve (Local)`

当用户拖拽手柄时：
1.  **获取鼠标位移**: 计算鼠标在世界空间中的移动量 `Total_Offset_World`。
2.  **逆向变换**: 使用缓存的 `parent_matrix.inverted()` 将世界位移转换回局部空间位移 `Total_Offset_Local`。
3.  **应用数据**: 将局部位移应用到 F-Curve 的 `handle_left` 或 `handle_right` 属性上。

这一过程确保了用户在世界空间的操作能正确映射回底层数据，即使物体也是在运动的父级下。

## 5. 总结
Motion Path Pro 的核心工作流程可以概括为：

1.  **Calculate (计算)**: 利用有限差分法，从复杂的父子层级关系中提取出真实的世界空间运动速度。
2.  **Visualize (可视化)**: 利用 GPU 批处理和广告牌技术，在 3D 视图中绘制出直观、恒定大小的交互手柄。
3.  **Interact (交互)**: 通过矩阵逆变换，将用户的世界空间操作精确映射回局部空间的 F-Curve 数据。

## 6. 路径线本体与自绘路径线的设计结论

### 6.1 Blender 原生路径线的角色与限制

- 当前插件依赖 Blender 原生的运动路径系统来生成并绘制那条**主路径线**，通过调用：
  - `object.paths_calculate / pose.paths_calculate`
  - `object.paths_update_visible`
- 这条线的绘制是 Blender 内部 C 端的 Overlay 代码完成的，Python 插件对其控制能力有限：
  - 可以控制：是否显示、帧范围、步长、部分颜色设定（取决于版本与主题设置）。
  - 无法直接控制：真正的像素级线宽、自定义渐变、虚线/描边/发光等复杂样式，也无法对原生线做几何上的“扭曲”或变形。
- 结论：**通过修改本插件代码，无法“直接”改变原生路径线的绘制风格，只能通过 Blender 自身的 Motion Path 设置做有限调整。**

### 6.2 插件自绘层：已经做了什么

- 在原生路径线之外，插件额外构建了一层“自绘可视化 + 交互”：
  - 自己计算关键帧处的世界空间位置和前/后一小段的速度向量。
  - 自己用 `gpu` 模块绘制：
    - 关键帧节点（圆形点 + 高亮效果）。
    - 手柄线段（与世界空间速度相切）。
    - 手柄控制点（小方块 + 选中高亮）。
  - 自己处理鼠标交互，将世界空间拖拽映射回 F-Curve。
- 因此，从视觉上看，当前系统实际上由两层叠加：
  1. 底层：Blender 原生路径线（负责“主轨迹”的存在感）。
  2. 上层：插件自绘的点 + 手柄 + 交互（负责“编辑体验”）。

### 6.3 如果改为插件自己绘制整条路径线，性能上的判断

需要拆开两个成本来看：

1. **数据计算成本**
   - 无论使用原生路径线还是自绘路径线，都需要“按帧”获取世界空间位置。
   - 当前插件已经通过 `build_position_cache` 完成了一份自己的采样缓存：
     - 使用 `frame_set` + `view_layer.update()` + `matrix_world` 获取每帧位置。
     - 同时用有限差分法计算前后速度。
   - Blender 原生 motion path 也有自己的一份内部缓存和计算流程。

2. **绘制成本**
   - 原生路径线的绘制在 C 端完成，效率极高。
   - 插件自绘线条使用 GPU 批绘制，在 Python 端只要顶点数据准备好，draw 调用本身也非常轻量。

综合结论：

- 若同时开启：
  - 原生路径线（Blender 自己算一遍位置）。
  - 插件位置缓存 + 手柄绘制（插件再算一遍位置）。  
  → 会有一定的**重复计算**，整体偏重。

- 若关闭原生路径线，仅使用插件的 `position_cache` 数据来自绘路径线：
  - 可以节省 Blender 内部那一份 motion path 计算与绘制。
  - 绘制开销非常小，主要成本来自 `build_position_cache` 的采样过程。
  - 性能上更合理的模式是：**“一份位置缓存，多处复用（手柄 + 自绘路径线）”。**

因此，性能上的合理策略是：

- 不追求“比原生快一个数量级”，而是通过**减少重复的按帧计算**，让整体系统更轻。
- 插件自绘路径线本身几乎不构成性能瓶颈，只要：
  - 顶点数据在关键帧改动时重建（用脏标记），而不是每帧重算。
  - 绘制时使用批处理，一次 draw 完成一整条或一组路径。

### 6.4 目前可选的“更快、更稳定”的绘制路线

在 Blender Python 插件层面，目前被广泛采用、且被官方文档和社区实践验证的高效绘制方式主要是：

- 使用 `gpu` 模块 + `batch_for_shader` 或 `GPUBatch`。
- 通过 `SpaceView3D.draw_handler_add` 注册绘制回调。
- 在回调内只做轻量工作，将已缓存的顶点数据批量提交给 GPU。

优化空间更多在“用法”而不是“更换 API”：

1. **缓存 + 脏标记**
   - 在关键帧或动画数据发生变化时重建：
     - `_state.position_cache`
     - 路径线顶点数组 / 批次对象
   - 绘制时不再做复杂逻辑，只是遍历已有 batch 做 draw。

2. **合批绘制**
   - 按对象或骨骼，将一整条路径（或一个角色的全部路径）合并为少量 batch。
   - 减少 draw call 次数，避免每段线单独绘制。

3. **避免在 draw 回调中做 `frame_set` / 依赖图更新**
   - 所有与时间采样相关的逻辑，应尽量放在：
     - 定时器驱动的 operator。
     - keyframe 编辑事件触发的更新逻辑。
   - draw 回调只消费“已经准备好”的数据。

小结：

- **API 层级上**，当前插件使用的 `gpu + draw_handler` 已经是 Blender Python 插件中主流且高效的绘制方案。
- **性能优化方向**，在于：
  - 统一位置缓存，避免原生路径与插件逻辑各算一遍。
  - 将未来的“自绘路径线”和现有手柄绘制建立在同一份缓存之上。
  - 用缓存 + 合批 + 轻量 draw 回调的模式组织代码。

## 7. 将路径线改为由插件自绘时的注意事项

### 7.1 架构层面的约束

- 单一数据源：
  - 路径线、关键帧点、手柄方向都应尽量共享同一份 `_state.position_cache` 与同一套帧范围配置。
  - 避免出现「线基于一套采样、手柄基于另一套采样」的情况，否则会出现视觉与交互上的不一致。
- UI 设计（第一阶段）：
  - 简化为一个核心布尔开关，例如“启用自定义路径线”。
  - 开关优先考虑放在动画相关面板下方的控制台 / 播放条区域，方便动画师在时间线上直接操作。
  - 启用后默认只使用插件自绘路径线，不再在 UI 中暴露原生路径线相关控制。

### 7.2 数据与采样策略

- 采样密度：
  - 当前 position_cache 只在关键帧处采样位置与速度，足以驱动手柄可视化。
  - 第一阶段自绘路径线直接采用「每一帧采样」的策略（步长为 1 帧），先保证行为可预测、实现简单。
- 采样时机：
  - 保持「只在数据变动时重算」的原则：
    - 关键帧编辑后；
    - 对象 / 骨骼绑定结构变更后；
    - 插件设置中与路径相关的参数（帧范围、步长）改变后。
  - 避免在 draw 回调中做任何 `frame_set` 或依赖图更新。

### 7.3 绘制实现细节

- 颜色与样式：
  - 可以根据以下维度设计着色：
    - 帧号（例如当前帧附近加亮）；
    - 速度大小（高速度区域加重颜色或改变色相）；
    - 选中状态（选中对象 / 骨骼对应路径加粗或变色）。
  - 注意与已有关键帧点、手柄颜色的协调，避免信息冲突。

### 7.4 与手柄可视化和交互的一致性

- 视觉一致性：
  - 第一阶段实现中，可以先只考虑子级在本地空间的运动（基于当前对象 / 骨骼的 location F-Curve），暂不处理父子级层级和约束带来的世界空间偏移。
  - 这样实现相对简单，便于快速完成自绘路径线的基础版本，后续再逐步引入基于 `matrix_world` 的世界空间轨迹与相切约束。
- 交互一致性：
  - 拖拽手柄后，必须触发 position_cache 和路径线顶点缓存的重建：
    - 确保用户拖动后路径线立即更新，避免“手柄动了、线没动”的错觉。
  - 若未来加入自绘路径线的直接拖拽（在路径线上拖点而非拖手柄），需要明确定义：
    - 该拖拽是反推关键帧位置，还是仅修改中间插值形状（后者会偏离现有 F-Curve 模型）。

### 7.5 当前阶段的取舍

- 不保留原生路径线逻辑：
  - 当自绘路径线方案稳定后，可以直接移除或停用与原生 Motion Path 紧耦合的代码与 UI，简化维护成本。
- 暂不针对大场景做专项优化：
  - 先以「每帧采样 + 简单绘制」为目标完成基础功能。
  - 若后续在真实项目中暴露出性能瓶颈，再专门设计采样步长、可见帧段等高级优化策略。
